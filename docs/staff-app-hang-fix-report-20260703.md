## ส่วนที่ 2 — รายละเอียด 7 ไฟล์: แก้อะไร / ถ้าไม่แก้เกิดอะไร / ช่วยอะไร

### 🆕 1. `utils/withTimeout.ts` — ไฟล์ใหม่ (~25 บรรทัด)

- **คืออะไร:** เครื่องมือกลาง "นาฬิกาจับเวลา" ครอบ promise — เกินเวลา = error ภาษาไทย
  "การเชื่อมต่อใช้เวลานานเกินไป กรุณาลองใหม่อีกครั้ง"
  ก๊อป pattern จาก `Callback.tsx` ที่พิสูจน์บน production แล้ว
- **ถ้าไม่มี:** ไม่มีมาตรฐานกลางกันค้าง
- **ช่วย:** "รอไม่มีที่สิ้นสุด" → "รอสูงสุด 8-15 วิ แล้วแจ้งผู้ใช้"

### 🔑 2. `pages/auth/Login.tsx` (+65) — ประตูทางเข้าของทุกคน

- **แก้:** เพดานเวลา liff.init 12 วิ / getProfile 10 วิ / loginWithLine 15 วิ / ตรวจลิงก์คำเชิญ 10 วิ + **ปุ่มใหม่ "ลองเชื่อมต่อใหม่"** เมื่อ LIFF ล้ม (พร้อมล้าง error เก่า)
- **ถ้าไม่แก้:** staff ขาประจำ (auto-login) เจอเน็ตสะดุด → ค้าง "กำลังเชื่อมต่อ..." ถาวร — **โอกาสโดนสูงสุดเพราะทุกคนผ่านจุดนี้ทุกเช้า** · LIFF ล้ม → ปุ่ม LINE ตายเงียบ → staff คิดว่าแอปพัง โทรหาแอดมิน
- **ช่วย:** เข้าไม่ได้รู้ตัวใน 15 วิ + ฟื้นเองได้ด้วยปุ่มเดียว

### 📡 3. `hooks/useStaffBookings.ts` (+111/-110) — ตัวอันตรายสุดเชิงระบบ (หน้า "ติดตาม GPS")

- **แก้ 7 อย่าง:**
  1. ลบ query DEBUG ที่ดึงงานพนักงานคนอื่น
  2. ลบ console.log 8 จุด
  3. Cache staffId ลด round-trip
  4. **Realtime ใส่ filter เฉพาะงานตัวเอง** (`staff_id=eq.<ตัวเอง>` ทั้งตาราง jobs และ staff_journeys — เช็ค schema แล้วคอลัมน์มีจริง)
  5. **Refetch เงียบ ไม่เด้ง spinner** (silent refresh)
  6. ชื่อ channel ไม่ซ้ำกันต่อคน (`staff-bookings-<id>`)
  7. Timeout ทุก query (10-15 วิ)
- **ถ้าไม่แก้:** สูตร N² — **10 คนเดินทาง × 10 คนเปิด Tracking = จอกะพริบทุกวินาที** · 30×30 = ~90 query หนัก/วินาที → **connection 60 เส้นเต็ม → ช้าลามทุกแอป ไม่ใช่แค่ staff**
- **ช่วย:** โหลดโตแบบ N ธรรมดา + จอนิ่ง + เร็วขึ้น (ตัด 1 query/รอบ)

### 🎛 4. `pages/StaffDashboard.tsx` (+12) และ 📄 5. `pages/StaffJobDetail.tsx` (+19)

- **แก้:** ปุ่ม **รับงาน / เริ่มงาน / จบงาน** ครอบเพดาน 15 วิ (ฟังก์ชันข้างในเดิมเป๊ะ) + ถ้า timeout → **ดึงสถานะจริงจาก DB มาเทียบอัตโนมัติ** กันเคส "คำสั่งถึง server แล้วแต่คำตอบเดินทางกลับมาไม่ทัน" → ไม่รับซ้ำ ไม่หลงสถานะ
- **ถ้าไม่แก้:** วินาทีที่ DB อืด staff ที่กดปุ่มพอดี **ติดกับกลางวงจรงาน** — รับ/จบงานไม่ได้ ปุ่ม disabled ถาวร → ลูกค้ารอ งานค้างสถานะ แอดมินต้องตามแก้มือ
- **ช่วย:** ปุ่มคืนสภาพใน 15 วิ ทำงานต่อได้เสมอ

### 🚗 6. `hooks/useGPSTracking.ts` (+79) และ 📍 7. `components/JobGPSControls.tsx` (+16)

- **แก้:** ทุก await ในปุ่ม **"เริ่มเดินทาง"** (rpc start_staff_journey 12 วิ / auth.getUser 8 วิ / อัปเดต jobs→traveling 10 วิ / ส่งพิกัดแรก 10 วิ) และ **"มาถึงแล้ว"** (ดึง journey 10 วิ / journey→arrived 10 วิ / jobs→arrived 10 วิ) + staff lookup 10 วิ — logic ลำดับสถานะไม่แตะเลย
- **ถ้าไม่แก้:** staff กดปุ่มบนถนน (เน็ตมือถือ = จุดสะดุดบ่อยสุด) → ค้าง "กำลังยืนยัน..." → **ติดสถานะ traveling ออกไม่ได้** → แผนที่ลูกค้าค้าง เวลาบริการเพี้ยน
- **ช่วย:** GPS ทุกขั้นหลุดแล้วกดใหม่ได้ทันที

แอปไม่ได้คุยผ่าน Vercel ตอนใช้งาน — Vercel แค่เสิร์ฟไฟล์ · อาการหมุนค้างทั้งหมดเกิดตอนแอปคุยกับ Supabase

### ผลตรงต่อ Supabase (เทียบกราฟที่วิเคราะห์)

| สิ่งที่เจอในกราฟ (7 วันก่อนแก้) | ไฟล์ที่แก้ | ผลหลัง push |
|---|---|---|
| WAL query กิน **63.1% ของเวลา DB** (1.29M calls → ส่ง event จริงแค่ 1,031) | ไฟล์ 3 (filter) | Subscription แคบลง → ตัวเลขนี้ลดชัดเจน ยิ่งคนเยอะยิ่งต่าง |
| Connections พีค **40-45/60** ทั้งที่คนน้อย | ไฟล์ 3 (ตัด DEBUG + เลิก N² + cache) | Query/คน/รอบ ลด ~40% → headroom กลับมา |
| Query jobs ช้า 1.9 วิ ถูกยิงซ้ำทุก event | เดิม N×M ครั้ง → เหลือเฉพาะ event ตัวเอง | โหลดสะสมหาย (ตัว query ยังช้าเท่าเดิม — งานเฟสหน้า) |
| เพดาน Realtime **500 connections / 500 msg/s** (Pro plan) | messages เดิมโต N×M → ชนตั้งแต่ ~70 คน | โตแบบ N → เพดานขยับไปหลักพัน |
| **Micro compute** ที่อัปเกรดฟรีแล้ว (RAM ×2, CPU dedicated) | — | เครื่องที่จ่ายแล้วถูกใช้กับงานจริง ไม่เผากับ query ขยะ |

### ผลกับ Vercel

- **ตอน push:** build เฉพาะ `the-bliss-at-home-staff` (~2 นาที) แบบ zero-downtime — แอปอีก 4 ตัว **ไม่ถูก rebuild** (diff อยู่ใน `apps/staff/` เท่านั้น)
- **Rollback:** Deployments → Promote เวอร์ชันก่อนหน้า = ย้อนกลับใน ~10 วินาที ไม่ต้องแตะ git
- **ความเร็ว:** ที่เทสว่า "เร็ว" บน tunnel คือ production build เดียวกับที่ Vercel จะเสิร์ฟ — ของจริงเร็วกว่าอีกเพราะ CDN edge
- **ข้อควรระวัง:** โปรเจกต์เคยมีปัญหา "Ignored Build Step" ข้าม deploy (commit `9aa6adb0` เคย force rebuild) → หลัง push ต้องตรวจ bundle บน prod ว่าขึ้นจริง

## ส่วนที่ 4 — เพดานรองรับ: ไม่ push vs push

| | ❌ ไม่ push | ✅ push |
|---|---|---|
| Staff ออนไลน์ทั่วไป | ~30-60 คน | **~300-400 คน** (ชนเพดาน Realtime 500) |
| หน้า GPS Tracking | **พังตั้งแต่ ~5-10 คน** เดินทางพร้อมกัน | ปกติจนถึงหลักร้อย |
| เมื่อระบบอืด | ค้างถาวร → refresh ถล่มซ้ำ → พังลูกโซ่ทุกแอป | เด้ง "ลองใหม่" → ระบบฟื้นเอง |
| ตัวกำหนดเพดาน | โค้ด (N²) | โครงสร้างพื้นฐาน — ขยายด้วยเงินได้ |

**ที่มาตัวเลข:**
- เอกสาร Supabase (ยืนยัน ณ วันที่ 3 ก.ค. 2026): Micro = 60 direct connections / 200 pooler / RAM 1GB · Pro Realtime = 500 concurrent connections / 500 messages/s (hard limit — ขยายได้ผ่าน support)
- กราฟจริงของโปรเจกต์ `theblissathome` (Database / Realtime / Query Performance reports)
- โค้ด production ปัจจุบัน (สูตร N²)
